## מטרת הפרויקט

מטרת הפרויקט היא לבנות תשתית ראשונית ל-Smart Organizational Knowledge Center: סוכן ידע שמאפשר לשאול שאלות בעברית על דוחות מחקר, ולקבל תשובות שמבוססות רק על מסמכי המקור.

בשלב הנוכחי הפרויקט מתמקד בדוח אחד מרכזי: Young Statistical Report. בהמשך ניתן להרחיב את המערכת למספר דוחות, סוגי מידע נוספים, וממשק משתמש מלא.

### 2026-05-01: הקמת בסיס RAG ראשוני במחברת

נבנתה מחברת הוכחת היתכנות ב-`data/rag_poc.ipynb`.

פעולות שבוצעו:

- טעינת PDF של Young Statistical Report.
- חילוץ טקסט לפי עמוד.
- שמירת metadata בסיסי: שם מקור ומספר עמוד.
- ניקוי טקסט עברי ותיקון בעיות RTL.
- חלוקה ל-chunks עם overlap.
- יצירת Embeddings באמצעות מודל רב-לשוני.
- בניית Chroma Vector DB מתוך הקטעים.
- שליפה סמנטית ראשונית באמצעות `similarity_search`.
- החזרת תוצאות עם מקור ועמוד.

תוצר:

- צינור RAG ראשוני ללא LLM.
- יכולת לשאול שאלה בעברית ולקבל קטעים רלוונטיים מהדוח.

## קבצים מרכזיים

`data/rag_poc.ipynb`

המחברת המרכזית של הוכחת ההיתכנות. היא כוללת את טעינת המסמכים, ניקוי הטקסט, חלוקה לקטעים, יצירת Embeddings, בניית Vector DB, שליפה סמנטית, סינון, דירוג, והרצת סוכן ידע בסיסי.

`app.py`

אפליקציית Streamlit אינטראקטיבית. הקובץ מציג ממשק בעברית לבחירת דוחות, כתיבת שאלה, קבלת תשובה מבוססת מקורות והצגת ציטוטים לפי עמוד.

`data/rag_agent_v0.py`

קובץ Python שמרכז את שכבת ה-Agent. הוא יושב מעל מנוע השליפה הקיים ומוסיף לוגיקה של זיהוי סוג שאלה, התאמת אסטרטגיית שליפה, דירוג תוצאות ויצירת תשובה מובנית בעברית.

`data/rag_index.py`

מודול הכנת המידע. הוא אחראי לחילוץ טקסט מה-PDF, חלוקה ל-chunks, שמירת קבצי ביניים, בניית Chroma DB וטעינת המאגר עבור האפליקציה.

`scripts/build_knowledge_base.py`

סקריפט בנייה ייעודי. מריצים אותו כדי להכין מראש את כל קבצי התמיכה ואת מאגר הווקטורים, כך שהאפליקציה תעלה מהר יותר.

`data/processed/`

תיקיית קבצי ביניים. כאן נשמרים `pages.jsonl`, `chunks.jsonl`, `metadata.csv` ו-`manifest.json` לכל אוסף דוחות.

`data/vector_db/`

תיקיית מאגרי Chroma. כאן נשמרים הווקטורים, האינדקס וה-metadata שמשמשים לשליפה בזמן שאלה.

`data/raw_reports/`

תיקיית מסמכי המקור. בשלב הנוכחי העבודה התחילה מדוח אחד, אך התיקייה כבר כוללת דוחות נוספים שניתן לשלב בהמשך.

`data/evaluation/`

תיקיית סט ההערכה. כאן נשמרים קובץ שאלות בלבד, קובץ תשובות צפויות בלבד, ובהמשך גם קובץ תוצאות ההרצה של המערכת.

## שלבי המחקר והפיתוח

### שלב 1: טעינת מסמכי PDF

נטענו קובצי PDF מתוך תיקיית `data/raw_reports`. הטקסט חולץ ברמת עמוד כדי לאפשר ציטוטים מדויקים בהמשך.

תוצר מרכזי:

- לכל עמוד נשמר טקסט.
- לכל קטע נשמרו נתוני מקור: שם הקובץ ומספר העמוד.

### שלב 2: עיבוד עברית וניקוי טקסט

בדוחות PDF בעברית מופיעות לעיתים בעיות חילוץ, כגון טקסט הפוך, סימנים משובשים, מספרים מעוותים ורווחים לא תקינים. לכן בוצעו פעולות ניקוי ייעודיות.

פעולות שבוצעו:

- תיקון כיוון טקסט עברי במקרים שבהם הטקסט חולץ הפוך.
- תיקון עיוותים במספרים, שנים ואחוזים.
- הסרת רעשים שנוצרו בתהליך חילוץ ה-PDF.
- האחדת רווחים ושבירות שורה.

### שלב 3: הכנת מידע ל-RAG

הטקסט חולק לקטעים כדי לאפשר שליפה ממוקדת. הקטעים כוללים חפיפה כדי לשמור על הקשר בין פסקאות סמוכות.

פעולות שבוצעו:

- חלוקה ל-chunks.
- הוספת overlap בין קטעים.
- סינון קטעים קצרים מדי.
- שמירת metadata לכל קטע.

### שלב 4: ייצוג סמנטי

נוצרו Embeddings באמצעות מודל רב-לשוני שמתאים גם לעברית. המטרה היא לאפשר חיפוש לפי משמעות, ולא רק לפי הופעת מילים זהות.

תוצאה:

- כל קטע טקסט קיבל ייצוג וקטורי.
- ניתן לשאול שאלות בעברית ולקבל קטעים קרובים מבחינה סמנטית.

### שלב 5: בניית Vector DB

נבנה מאגר וקטורי באמצעות Chroma. המאגר מאפשר חיפוש סמנטי מהיר על גבי הקטעים.

תוצר:

- `vector_db` במחברת.
- חיפוש סמנטי באמצעות `similarity_search`.
- החזרת תוצאות עם metadata של מקור ועמוד.

### שלב 6: שיפור איכות השליפה

לאחר בדיקות ראשוניות, נוספו כללי סינון ודירוג כדי לשפר את איכות התוצאות.

שיפורים שבוצעו:

- שילוב בין חיפוש סמנטי לבין התאמת ביטויים מדויקים.
- סינון טבלאות כאשר המשתמש מבקש הסבר מילולי.
- סינון קטעים חלשים כמו מבואות כלליים, תוכן עניינים או הגדרות לא רלוונטיות.
- דירוג גבוה יותר לקטעים עם נתונים, אחוזים, שנים או מונחים מרכזיים כמו ישראל ו-OECD.

### שלב 7: שכבת תשובה בסיסית ללא LLM

בשלב זה לא נעשה שימוש במודל שפה לצורך ניסוח תשובות חופשיות. במקום זאת, המערכת מחזירה קטעים רלוונטיים מתוך המקור עצמו.

יתרונות:

- הקטנת סיכון להמצאת מידע.
- שמירה על נאמנות גבוהה למסמכי המקור.
- שקיפות מלאה לגבי המקורות.

מגבלות:

- התשובה אינה תמיד מנוסחת בשפה טבעית מלאה.
- יש תלות באיכות חילוץ הטקסט מה-PDF.
- שילוב מידע ממספר מקורות עדיין בסיסי.

### שלב 8: בניית סוכן ידע זמני

נוסף קובץ `data/rag_agent_v0.py`, שמגדיר סוכן ידע בסיסי מעל מנוע השליפה.

יכולות הסוכן:

- קבלת שאלה בעברית.
- זיהוי סוג השאלה:
  - הגדרה
  - נתונים
  - סיכום
  - השוואה
- התאמת אסטרטגיית השליפה לפי סוג השאלה.
- שליפה סמנטית מתוך `vector_db`.
- סינון ודירוג תוצאות.
- החזרת תשובה מובנית בעברית.
- הצגת מקורות לפי עמוד.

## מצב נוכחי

המערכת הנוכחית היא Knowledge Retrieval Agent בגרסת v0.

היא כוללת:

- טעינת PDF.
- חילוץ טקסט לפי עמוד.
- ניקוי טקסט עברי.
- חלוקה לקטעים.
- יצירת Embeddings.
- Vector DB.
- שליפה סמנטית.
- סינון ודירוג.
- Agent בסיסי.
- תשובה עברית עם מקורות לפי עמוד.

היא עדיין לא כוללת:

- ניסוח תשובות מתקדם באמצעות LLM.
- ממשק משתמש.
- הערכה שיטתית.
- תמיכה מלאה במספר רב של דוחות.
- ניהול הרשאות, גרסאות ומעקב אחר מקורות ברמת מערכת ארגונית.

## מגבלות ידועות

- איכות התשובה תלויה באיכות חילוץ הטקסט מה-PDF.
- טבלאות מורכבות אינן מפוענחות בצורה מלאה.
- יש עדיין מקרים שבהם טקסט עברי מחולץ עם עיוותי קידוד או כיוון.
- ללא LLM, התשובה מבוססת בעיקר על הצגת קטעים ולא על סינתזה מלאה.
- אין עדיין מדדי איכות פורמליים לשליפה ולתשובה.

## שלבים מומלצים להמשך

### 1. בניית סט הערכה

לבנות 20-30 שאלות benchmark בעברית, מסוגים שונים:

- שאלות הגדרה.
- שאלות נתונים.
- שאלות השוואה.
- שאלות סיכום.
- שאלות עם תשובה שמופיעה בכמה עמודים.

לכל שאלה כדאי להגדיר:

- תשובה מצופה.
- עמודי מקור נכונים.
- רמת קושי.
- סוג השאלה.

### 2. שיפור שכבת התשובה

להוסיף יכולת סינתזה זהירה של תשובות, עם או בלי LLM.

דרישות חשובות:

- לא לענות מעבר למה שנמצא במקורות.
- לציין כאשר אין מספיק מידע.
- להפריד בין עובדות, השוואות ותובנות.
- להציג מקורות לכל טענה מרכזית.

### 3. הרחבה למספר דוחות

להרחיב את המאגר מ-1 דוח ל-8 דוחות קיימים בתיקייה.

נדרש:

- metadata עשיר יותר לכל דוח.
- הפרדה בין סוגי מסמכים.
- אפשרות לסנן לפי דוח, שנה או תחום.
- בדיקת כפילויות בין מקורות.

### 4. טיפול טוב יותר בטבלאות

מכיוון שדוחות סטטיסטיים כוללים טבלאות רבות, נדרש מנגנון ייעודי לטבלאות.

אפשרויות:

- חילוץ טבלאות לכלי מובנה כמו pandas.
- שמירת טבלאות כמבנים נפרדים מהטקסט.
- שילוב בין שליפה סמנטית לבין חיפוש ערכים מספריים.
- הצגת תשובות מספריות עם מקור מדויק.

### 5. בניית ממשק משתמש

לבנות ממשק פשוט ב-Streamlit או Chainlit.

יכולות מומלצות:

- תיבת שאלה בעברית.
- הצגת תשובה.
- הצגת מקורות לפי עמוד.
- אפשרות לפתוח את קטע המקור.
- בחירת מסמך או קבוצת מסמכים.

## החלטות תכנון מרכזיות

- בשלב הראשון הוחלט לא להשתמש ב-LLM כדי לבנות בסיס אמין ושקוף.
- השליפה מבוססת Embeddings כדי לאפשר חיפוש לפי משמעות בעברית.
- ציטוטים נשמרים ברמת עמוד, משום שזה מספיק ברור לשימוש מחקרי ראשוני.
- קוד ה-Agent הופרד לקובץ Python עצמאי כדי שהמחברת תישאר קריאה.
- המחברת כוללת הסברים בעברית כדי להתאים להצגה לבעלי עניין לא טכניים.

## סיכום

הפרויקט נמצא בשלב הוכחת היתכנות מתקדם. כבר קיימת תשתית RAG מלאה ללא LLM, עם שליפה סמנטית בעברית, דירוג תוצאות, וסוכן ידע בסיסי שמחזיר תשובות עם מקורות לפי עמוד.

השלב הבא החשוב ביותר הוא הערכה שיטתית: בניית סט שאלות, בדיקת איכות השליפה, ולאחר מכן שיפור שכבת התשובה והרחבה למספר דוחות.

## עדכון יומן מחקר: חיבור ראשוני ל-OpenAI API

### מטרת העדכון

לאחר שנבנה בסיס הערכה הכולל 128 שאלות תיקוף עם תשובות צפויות, הוחלט להתקדם לשכבת ניסוח תשובות בעזרת LLM. ההחלטה היא לשמור על מצב בסיסי ללא LLM, ובמקביל להוסיף מצב מתקדם שמנסח תשובה עברית על בסיס המקורות שנשלפו בלבד.

### קבצים שנוספו או עודכנו

- `.env` - קובץ מקומי למפתח OpenAI. הקובץ מכיל כרגע מפתח דמה בלבד ואינו מיועד ל-GitHub.
- `.env.example` - תבנית בטוחה לשיתוף, בלי מפתח אמיתי.
- `data/rag_llm_answer.py` - מודול חדש לניסוח תשובות בעזרת OpenAI API.
- `app.py` - נוסף מצב בחירה בין תשובה בסיסית ללא LLM לבין תשובה מנוסחת עם LLM.
- `requirements.txt` - נוספו הספריות `openai` ו-`python-dotenv`.

### לוגיקת ה-LLM שנוספה

המודול החדש מקבל את תוצאת ה-Agent הקיים, כולל השאלה, המקורות והעמודים שנשלפו, ושולח ל-LLM רק את המידע הזה. ההנחיות למודל מחייבות:

- תשובה בעברית בלבד.
- שימוש רק במידע שמופיע במקורות שנשלפו.
- ציטוטי עמוד לכל טענה עובדתית.
- הודעה מפורשת כאשר אין מספיק מידע במקורות.
- שמירה על ניסוח קצר, מקצועי וברור.

### בדיקות איכות לתשובת LLM

באפליקציה נוספה תצוגת בדיקות בסיסיות עבור תשובה מנוסחת:

- האם התשובה כוללת עברית.
- האם קיימים מקורות שנשלפו.
- האם התשובה כוללת ציטוטי עמוד.
- האם העמודים שצוטטו מופיעים בתוך המקורות שנשלפו.
- האם הופעלה הנחיית שימוש במקורות בלבד.

### הגדרת מפתח API

כדי להפעיל תשובה מנוסחת עם LLM יש לעדכן את הקובץ `.env` כך:

```env
OPENAI_API_KEY=להדביק_כאן_מפתח_אמיתי
OPENAI_MODEL=gpt-5.4-mini
OPENAI_MAX_OUTPUT_TOKENS=900
```

חשוב: אין להעלות את `.env` ל-GitHub. לשיתוף הפרויקט משתמשים רק ב-`.env.example`.

### מצב נוכחי לאחר העדכון

המערכת כוללת כעת שני מצבי עבודה:

- מצב בסיסי ללא LLM: משתמש בשליפה, סינון ודירוג ומחזיר תשובה מבוססת מקורות.
- מצב מנוסח עם LLM: משתמש באותם מקורות שנשלפו, ומנסח תשובה עברית ברורה עם ציטוטי עמוד.

עדיין לא בוצעה קריאת API אמיתית משום שהמפתח בקובץ `.env` הוא מפתח דמה. לאחר הכנסת מפתח אמיתי, יש לבדוק את המצב החדש מול שאלות התיקוף ולשמור תוצאות הערכה נפרדות ל-LLM.

## עדכון יומן מחקר: תיקון סביבת הפריסה ב-Streamlit Cloud

### הבעיה

בפריסה הראשונה ב-Streamlit Cloud הופיעה שגיאת טעינה בזמן ייבוא Chroma. לפי ה-Traceback, Streamlit הריץ את האפליקציה על Python 3.14, ונוצרה שגיאת תאימות בין `chromadb`, `opentelemetry` ו-`protobuf`.

### הפעולה שבוצעה

נוספו הגדרות סביבת הרצה יציבות יותר:

- נוסף קובץ `runtime.txt` עם Python 3.11.
- עודכן `requirements.txt` עם `protobuf==3.20.3`.

### הסיבה

Python 3.11 היא סביבת הרצה יציבה יותר עבור Chroma, LangChain ו-Sentence Transformers. קיבוע `protobuf` לגרסה 3.20.3 מונע שגיאת descriptor ידועה שמופיעה בחלק משילובי הספריות.

### בדיקה נדרשת

לאחר העלאת השינוי ל-GitHub, יש לבצע Reboot או Redeploy ב-Streamlit Cloud ולבדוק שהייבוא של Chroma עובר בהצלחה.
